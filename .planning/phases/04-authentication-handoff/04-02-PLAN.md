<objective>
Complete the authentication lifecycle with **`archeo clear-session`** and lock down the
**AUTH-03 hygiene guarantees**. `clear-session <url|hostname>` deletes the persisted per-
hostname profile (or the whole profiles root with `--all`), idempotently, with a **path-escape
refusal** so a crafted hostname can never steer `rmSync` outside `.archeo/profiles/` (D4-05).
The hygiene suite proves the persisted session is a single gitignored location that never leaks
into the capture store or the spec (AUTH-03).

Purpose: AUTH-03 (the persisted session lives in one gitignored local location, never enters
the capture store or spec, and is cleared on request) is delivered here as tested code — a safe,
idempotent, containment-checked deletion plus the standing hygiene assertions.

Output: new `src/cli/clearSession.ts` (deletion + containment guard, reusing 04-01's
`sanitizeHostname`), the `archeo clear-session [target] --all` subcommand in `src/cli/index.ts`
(gate-free, D4-05), unit tests for deletion/idempotency/path-escape, and a standing AUTH-03
hygiene suite. `.gitignore` already covers `.archeo/` (verified) — this plan pins that coverage
with a test and does NOT edit `.gitignore`.
</objective>

  // = path.resolve(profilesRoot ?? PROFILES_ROOT, sanitizeHostname(hostname)); then assert the

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| profile dir ↔ git | The persistent profile holds live session cookies; it must never be committable |
| clear-session target ↔ filesystem | A user-supplied hostname/URL drives an rmSync path; a crafted value must not delete outside .archeo/profiles/ |
| profile dir ↔ capture store / spec | The profile path and its contents must never appear in a capture record or the generated spec |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-06 | Information Disclosure | live session cookies committed to git | mitigate | .archeo/ is gitignored (T-02-05); .archeo/profiles/ is covered. The hygiene suite pins `.archeo/` in .gitignore so it cannot be silently un-ignored (AUTH-03). |
| T-04-07 | Elevation of Privilege / Tampering | clear-session path traversal deleting arbitrary dirs | mitigate | resolveProfilePath sanitizes the hostname (04-01) AND containment-checks the resolved path against the resolved profiles root before rmSync; a path that escapes throws → exit 1. Unit-tested with '../../etc'. Defense in depth (two independent guards). |
| T-04-08 | Information Disclosure | profile path / content in capture store or spec | mitigate | browser.ts passes profileDir only to launchPersistentContext (a playwright arg); the store is scoped by hostname, not profile path; the spec derives only from capture.jsonl. Hygiene suite greps store code paths and a generated spec for the profiles path — must be absent. |
| T-04-09 | Denial of Service (self) | `--all` deleting every profile | accept | Intentional user-directed destructive local action; no gate per D4-05 (destroys local state only, opens no browser). The command prints exactly what it deleted. |
| T-04-10 | Tampering | clear-session accidentally invoking the gate/browser | mitigate | Source-inspection assertion: the clear-session action contains no runAuthorizationGate / openAndWait / openForLogin (D4-05 gate-free guarantee). |

No package installs in this plan — no supply-chain checkpoint required. No outbound network surface added.
</threat_model>
