<objective>
Ship the safety floor's **one sanctioned bypass** and the CAP-06 **redaction seam** — both loud,
both opt-in, both careful. `--allow-writes` disables **only** the hold behaviour: mutations now
reach the server and are captured `held:false`, with an `allowWrites` flag stamped into the
manifest and the spec coverage so a consumer knows the writes were real. Everything else that
makes the floor safe **stays**: the destructive-GET prompt still fires, redaction is untouched
(CAP-05), and in explore mode the never-click blocklist still applies. Turning it on is
deliberately hard to do by accident: an unmissable multi-line red banner + a mandatory extra
confirmation, and in non-TTY it refuses unless the scriptable companion flag `--i-accept-writes`
is present too. The CAP-06 enhancement is **scope-cut per D6-07** to an external-command seam —
`--redaction-model <cmd>` pipes redacted-candidate JSON to a user command and applies whatever
extra field paths it returns; a no-op default and a tiny example script ship in the artifacts. The
seam can only **add** redactions and fails **closed**, so the floor remains the guarantee.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| --allow-writes ↔ live server | This is the floor's ONE sanctioned bypass: with it on, mutations really hit the user's own account. It must be impossible to enable by accident — an unmissable banner + an explicit confirmation, and non-TTY refuses without the companion flag. The destructive-GET tripwire and redaction do NOT relax. |
| CAP-06 external command ↔ arbitrary code execution | --redaction-model runs a user-supplied command. It is the user's own machine and their own flag (same trust as any CLI argument); it runs with a timeout, fails closed to [], and can only ADD redactions — it can never weaken the floor or exfiltrate (it only receives already-base-redacted candidate JSON). |
| allowWrites flag ↔ consumer trust | A consumer of the spec must be able to tell whether the captured writes were real; the manifest + spec coverage carry allowWrites so a "writes were real" run is never mistaken for a held-floor run. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation |
|-----------|----------|-----------|-------------|------------|
| T-06-15 | Tampering (unintended real mutations) | --allow-writes | mitigate | Off by default; enabling requires a banner + an extra y/N; non-TTY refuses without --i-accept-writes; the destructive-GET tripwire still prompts; the never-click blocklist still applies. Live-proven in 06-06 (ledger 1 under flag, ledger 0 on the next default run). |
| T-06-16 | Elevation of Privilege | user-supplied --redaction-model command | mitigate/accept | The command is the user's own (their machine, their flag); it runs with a 2s timeout, gets only base-redacted candidate JSON, and can only ADD field redactions (applyExtraRedactions never re-exposes). Failure fails closed to the base floor. Documented in the seam README. |
| T-06-17 | Information Disclosure | writes captured under --allow-writes leaking secrets | mitigate | Redaction (CAP-05) is UNCHANGED under --allow-writes — captured real-write records are redacted exactly as reads are; the redactionHook only strengthens this. |
| T-06-18 | Repudiation | a real-write run mistaken for a held-floor run | mitigate | manifest.allowWrites + coverage.allowWrites mark the session mode so the provenance is unambiguous. |

No package installs; `node:child_process` is the only new built-in (not a network client). The no-network guard stays green.
