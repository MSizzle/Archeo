<objective>
Ship `archeo compare <urlA> <urlB>` (VALID-01) as a **thin orchestration wrapper** and prove the
two-target architecture has **no duplicated codepaths** (VALID-02). The divergence engine
(`diffSpecs`/`formatDriftTable`) and the URL-accepting exploration layer (`runExplore`) already
exist — `compare` runs the **same exploration configuration** against A then B, each fully
isolated, then reuses `diffSpecs` and renders an **original-vs-rebuild divergence report** plus a
`compare-report.json`. It does **not** reimplement the diff, the interceptor, or the explore loop.

Two design realities discovered during context-gathering shape this plan and MUST be honored:
1. **`runExplore` calls `process.exit(0)`** (in `gracefulShutdown`) — two in-process runs are
   impossible. Production explores each target in its **own child process running the shipped
   `archeo explore` command** (the exact shared `runExplore`→`generateSpec` path). This *also*
   makes VALID-02's "no duplicated codepaths" the strongest possible (separate process + store +
   profile), and keeps `compare.ts` free of any capture/explore logic.
2. **Both dogfood targets share hostname `localhost`** (only ports differ) and profiles are
   per-hostname — so each target must run under a **distinct isolated run root** to prevent
   cross-contamination.

The orchestration takes an **injected per-target runner** so it is unit-testable without a browser
(fake runner writes fixture specs; the real diff runs). Output: `src/cli/compare.ts`, the CLI
registration in `src/cli/index.ts`, and two test files. No floor/redaction/drift change.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| compare explores TWO live targets ↔ floor | A rebuild is still someone's live app. Both targets MUST stay floor-ON; compare registers no write-enabling flag and passes none to either child. Pinned by the floor-ON structural assertion. |
| divergence report ↔ trust | A false positive (reporting divergence where none exists) erodes trust in the tool. diffSpecs is already zero-false-positive on identical input; formatDivergence must not add spurious findings, and the determinism caveat foregrounds the reliable backend-contract signal over frontier-dependent page/flow noise. |
| two-target isolation ↔ cross-contamination | Two same-hostname targets (localhost:portA vs :portB) would share a per-hostname profile and (without isolation) a capture store. compare runs each target under a DISTINCT isolated run root; the unit test asserts the two run roots differ. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation |
|-----------|----------|-----------|-------------|------------|
| T-08-01 | Tampering (a write reaches a live rebuild) | compare's two exploration runs | mitigate | Floor ON both runs; no --allow-writes option on compare; structural test asserts no write-enabling token in compare.ts; each child runs the shipped floor-ON explore path. |
| T-08-02 | Information Disclosure / Repudiation (false divergence) | formatDivergence / compare-report | mitigate | Reuse diffSpecs (zero false positives on identical input); formatDivergence adds no computation; determinism caveat steers interpretation to the stable signal. |
| T-08-03 | Tampering (cross-target state bleed) | isolated run roots | mitigate | Distinct isolated run root per target (separate .archeo/captures + .archeo/profiles); unit test asserts distinct roots — safe even for same-hostname targets. |
| T-08-04 | Elevation / Spoofing (duplicated bespoke capture path drifts from the floor) | compare vs shipped explore | mitigate | compare defines NO capture/explore logic; it invokes the shipped `archeo explore` path; VALID-02 structural proof pins this. |

No package installs; no new outbound surface; the no-network guard is re-run as part of the suite.
</threat_model>
