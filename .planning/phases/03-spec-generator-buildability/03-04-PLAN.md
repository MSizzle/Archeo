<objective>
Close the value loop (BUILD-01): prove an Archeo spec produced from manually captured traffic is
consumable by a real, separate coding agent that never sees the target's source. Reuse the Phase 2
(02-04) live target app + auto-firing-page harness technique to drive a real scripted capture through
the unmodified CLI, generate the spec with `archeo spec`, hand ONLY that spec to a fresh builder
agent, and verify the resulting rebuild runs and serves the spec's endpoints. A scored
`03-04-BUILDABILITY.md` records the outcome. Per user directive this checkpoint is autonomous (as
02-04 was): no human verification step.

Purpose: BUILD-01 — a builder agent produces a runnable approximation of the target from the spec
alone, confirming the spec is valuable before autonomy (Phase 5) is invested.

Output: a capture driver + a rebuild scorer (under the phase's `03-04-buildability/` dir) and the
buildability report. NO changes to `src/` or `test/`.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| builder agent → target-app source | The "spec-only" claim collapses if the builder can see the target app or repo source |
| scripted capture → real CLI | The proof is only meaningful if it runs the unmodified production pipeline (as 02-04 established) |
| harness scripts → src/ GATE-03 guard | Harness .mjs files use node:http as a client — they must live under .planning/, never under src/ |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-13 | Tampering (proof integrity) | builder sees more than the spec | mitigate | The builder receives ONLY archeo-spec.json in an isolated input dir; the isolation is attested in the SUMMARY + report and is auditable (input dir contents listed). A fresh orchestrator-spawned agent has no repo context. |
| T-03-14 | Tampering (proof integrity) | mock pipeline instead of the real CLI | mitigate | capture-driver.mjs spawns the unmodified `node src/cli/index.ts … --i-have-authorization`; the CLI's real attestation + destructive prompt on stdout prove it is the production entrypoint (mirrors 02-04). |
| T-03-15 | Information Disclosure | GATE-03 guard tripped by harness http client | accept | Harness scripts live under .planning/ (not src/); the no-network guard scans src/ only, so node:http client use in the harness is out of scope of the guarantee (the guarantee is about shipped tool code, not proof scaffolding). |
| T-03-16 | Information Disclosure | live secrets in the produced spec | mitigate | The 02-04 target app plants only fake secrets; Phase 2 proved redaction fails closed and no secret reaches disk. The spec is derived from the already-redacted store, so it carries type names, not values (re-checkable by grep in the report). |

No package installs in this plan — no supply-chain checkpoint required. No changes to shipped src/ or test/.
</threat_model>
