<objective>
Add the two remaining safety signals. First, the destructive-GET tripwire (FLOOR-04): a GET whose path contains a destructive token is held and requires an explicit terminal y/N confirmation before it can fire — so a "delete via link" cannot auto-execute against the user's live account. Second, the dead-end signal (FLOOR-07 / D-05, detect+record only): when the app errors past a held write, that state is recorded as a dead end linked to the held write, for the Phase 5 agent loop to consume — no backtracking is built here.

Purpose: Close the floor. After this plan a real user is protected from destructive GETs and the store carries the dead-end signal that later autonomy depends on.

Output: extended `classifier.ts` (destructive-token detection), `interceptor.ts` (stdin confirmation + dead-end detection), `store.ts` (dead-end helper), and extended test suites.
</objective>
