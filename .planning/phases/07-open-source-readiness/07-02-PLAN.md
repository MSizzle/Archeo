|----------|-------------|
| shipped example bytes ↔ the target's real secrets | Examples are published to a public repo. A leaked token/cookie/email in a spec would be a real disclosure. The secret-clean grep gate over examples/ is the mitigation — proving redaction on exactly the bytes that ship. |
| CONTRIBUTING scope statement ↔ project posture | An unclear scope statement invites contributions that weaken the floor or add telemetry. The explicit IN/OUT list is the guardrail. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation |
|-----------|----------|-----------|-------------|------------|
| T-07-03 | Information Disclosure | examples/ shipped specs | mitigate | Secret-clean grep gate over every example (Task 2); real generator output only, no hand-scrubbing (which could mask a redaction bug); reject any spec with a real value. |
| T-07-04 | Repudiation / provenance | examples/ | mitigate | Each example carries its exact generating command + origin (public demo vs verification fixture) so a reviewer can reproduce and trust it. |
| T-07-05 | Tampering (scope creep) | CONTRIBUTING scope | mitigate | Explicit OUT-of-scope list (bundled models, telemetry, floor-weakening defaults, cloning framing) states what will be rejected. |

No package installs; no new outbound surface; no src/ change; the no-network guard is re-run.
